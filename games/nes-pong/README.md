# LWX Pong (NES)

A tiny CC0 NES game for LibretroWebXR test content. Single screen, NROM
(mapper 0). Left paddle = player (D-pad up/down — the WebXR frontend's RetroPad
maps to the NES controller); right paddle is a CPU that tracks the ball. Score
is shown as stacked blocks on each side.

This is the **cc65 + neslib template-fill** workflow from
`docs/research/nes-game-creation.md`: the timing-critical hardware code is frozen
in the vendored boilerplate, and the only file we author is `main.c` (game logic
calling neslib's documented API). That split is what makes AI-authored NES games
reliable.

## Build

```
node scripts/make-nes-pong.mjs      # from the repo root
```

Needs **cc65** installed. The script finds `cl65` via `%CC65_HOME%\bin`, then
`C:\cc65\bin`, then `PATH`. It generates `tiles.chr` (tile 0 blank, tile 1 = a
solid 8×8 block — all the art Pong needs), builds with
`cl65 -t nes -C nes.cfg main.c crt0.s`, and writes
`public/roms/freeware/lwx-nes-pong.nes` (NROM-256: 32K PRG + 8K CHR).

## Files

| File | Origin | License |
|---|---|---|
| `main.c` | **ours** — the game | CC0 |
| `nes.cfg` | ours (adapted from clbr/nes cnrom) | CC0 |
| `crt0.s`, `neslib.sinc`, `neslib.h`, `display.sinc`, `famitone2.sinc` | neslib (Shiru, fork by clbr) — frozen boilerplate | zlib (see `NESLIB-COPYING.txt`) |
| `tiles.chr` | generated by the build script | CC0 |

The compiled ROM is **CC0** — neslib's zlib license does not taint the output;
keep `NESLIB-COPYING.txt` for the vendored library source.

## Known quirk

The ROM renders correctly on **fceumm** (the core the manifest pins for it) and
in most emulators, but **nestopia shows a black screen** for this particular
ROM. The manifest entry therefore sets `"core": "fceumm"`.
